## Work Completed

### Global watermark and fleet-wide ingest model written into project notes
- Added **Global ingest cursor, backfill, and retention** to `project-notes.md`.
- Defined watermark \(W = \max(\text{observed\_at})\) across **all** stored rows as the **single progress clock** for global pulls (not per-station cursors for scheduling).
- **Why:** Realtime jobs are designed to hit **shared windows** for the whole marine list; one global maximum matches “have our scheduled polls advanced through time?” Sparse stations **without** readings in a chunk do not define \(W\)—that is explicit so backfill logic is not mistaken for per-station gap repair.
- **Outcome:** Implementers have an unambiguous rule for lag detection and backfill scope.

### Steady-state vs recovery modes
- Documented **steady state:** frequent runs (for example **every minute**) with an **overlapping** fetch of roughly the **last ~2 minutes**, preserving idempotent dedupe with minute-normalized keys.
- Documented **recovery:** when **`now − W` ~> 1.5 minutes**, enter **backfill**, advancing \(W\) toward `now` in **15-minute** chunks—the same safe bulk granularity already validated against GeoMet limits in reference tooling.
- **Why:** Small overlap minimizes missed rows under jitter; **15-minute** chunks bound each recovery request size on the upstream API while clearing multi-hour or multi-day downtime gaps predictably.
- **Outcome:** Clear state machine narrative for Nest scheduler design: normal tick vs catch-up loop.

### Cold start and bounded history
- Documented **empty DB:** walk forward from **`now − 1 year`** in **15-minute** steps until caught up (maximum depth aligned with longest UI horizon).
- Documented **rolling retention:** scrap observations older than **one year** so Postgres stays **bounded on Railway** and matches the **largest selectable history** in the UI—storage tracks product contract, not accidental hoarding.
- **Why:** Portfolio deployment favors **predictable disk**; retaining beyond UI need adds cost without user value. One-year cap is also an easy story in interviews: **schema + ingest + UX window line up**.
- **Outcome:** Storage growth has a closed-form upper bound tied to station count × cadence × one-year horizon (plus indexes), not “whatever we ingested forever.”

### Operational implementation hint
- Noted retention should prefer **daily batch or partitions** over deleting on every minute tick—**why:** avoid thrashing IO and long locks on Railway Postgres during hot paths.
- **Outcome:** Safer path to production without coupling retention cost to ingest frequency.

## Technical Decisions and Trade-offs

- **Decision:** Use **global** \( \max(\text{observed\_at}) \) rather than per-station high-water marks for **scheduling** backfill.  
  **Alternatives considered:** Per-station lag triggers; min timestamp across stations.  
  **Why chosen:** Ingest jobs are **fleet-wide window calls**; the watermark answers “how far forward has global polling reached?” Per-station silence is a **data characteristic**, not necessarily a missed poll window.

- **Decision:** **~1.5 minute** lag threshold to declare “behind” relative to \(W\).  
  **Alternatives considered:** Longer thresholds only; station-specific thresholds.  
  **Why chosen:** Matches minute cadence plus small tolerance; tunable constant once live metrics exist.

- **Decision:** **15-minute** recovery chunks (same as validated bulk sizing).  
  **Alternatives considered:** Hourly chunks (fewer calls but coarser recovery steps); single giant catch-up calls (risk hitting **10,000-feature** paging limits).  
  **Why chosen:** Aligns with known GeoMet constraints and existing script semantics.

- **Decision:** **One-year rolling retention** tied to UI max range.  
  **Alternatives considered:** Indefinite retention; retention shorter than UI (would contradict product).  
  **Why chosen:** Cost discipline and coherent product–storage contract.
