## Technical Decisions and Trade-offs

- **Decision:** Use ECCC MSC GeoMet as the single API source for now.  
  **Alternatives considered:** Mixed-source ingestion (NDBC + Canadian feeds) from day one.  
  **Why chosen:** One-source ingestion reduces integration complexity and keeps early architecture cleaner while still covering Canadian marine goals.

- **Decision:** Treat `swob-marine-stations` as station registry and `swob-realtime` as observation stream.  
  **Alternatives considered:** Assuming station metadata and measurements come from one endpoint.  
  **Why chosen:** The split model matches actual API design and avoids schema confusion during implementation.

- **Decision:** Accept time-window chunking for broader pulls instead of forcing single-call completeness.  
  **Alternatives considered:** Single-call weekly/hourly bulk retrieval.  
  **Why chosen:** The 10,000-feature response cap makes large single-call pulls incomplete; controlled chunking preserves data completeness with low operational complexity.

## Problems Encountered

- **Problem:** Large realtime windows produced files too large to comfortably format/view in editor.  
  **Diagnosis:** 24h and 1h calls both hit the 10,000-feature page cap and generated very large JSON payloads.  
  **Resolution:** Reduced window to 15 minutes for complete, manageable inspection and retained larger files as reference artifacts.

- **Problem:** Initial assumption that one feature equals one station in realtime output.  
  **Diagnosis:** Feature counts far exceeded marine station count; repeated station IDs appeared across timestamps.  
  **Resolution:** Confirmed realtime features are observation rows and documented station-grouping logic by `wmo_synop_id-value`.
