- **Outcome:** Implementation has a single source of truth for first migration and Nest ingest mapping work.

### Domain reality used to constrain schema (not generic “best practice”)
- Confirmed that Canadian marine buoy **operational** data is typically **minute-level** (or slower) for the most frequent stations—not millisecond-paced instrument streams.
- Confirmed use case: **field decision support** (map + trends + thresholds), not audit-grade reprocessing of corrected laboratory series.
- **Why:** Structural choices should match **actual delivery cadence and correction expectations**. Planning for millisecond uniqueness or revision chains would add columns and logic that this product will never exercise; that is **overengineering relative to the data producer and the user question** (“are conditions viable for work?”).
- **Outcome:** Minute-normalized `observed_at` plus discard/upsert on duplicate `(station, minute)` is the intentional identity model, not a temporary shortcut.

### Clarified optional metadata vs MVP lean
- **`ingested_at`:** kept as useful because it separates **sensor time** from **pipeline success time** (health, lag) without storing raw payloads.
- **Provenance / upstream `Feature.id`:** omitted from v1 schema; dedupe relies on `(station_wmo_id, observed_at)` only.
- **Why:** Overlap fetches can deliver the same logical reading twice; minute-bucket dedupe addresses that. Storing every upstream feature id mainly helps forensic traceability—valuable for some systems, **not required** for CTI’s stated MVP or for buoy data that is not hand-edited row-by-row after the fact.
- **Outcome:** Simpler rows and fewer indexes; add provenance later only if real duplicate or ambiguity appears in production-like runs.

## Technical Decisions and Trade-offs

- **Decision:** Normalize `observed_at` to **minute precision** for uniqueness and dedupe.  
  **Alternatives considered:** Full ISO timestamps with millisecond resolution; separate provenance keys for every row.  
  **Why chosen:** Buoy updates are not produced at millisecond granularity for this use case; sub-minute identity adds complexity without improving map or 24h/7d charts. Millisecond planning would be **disconnected from the physics and the upstream product**, i.e. overengineering.

- **Decision:** **No** upstream GeoJSON `Feature.id` column in v1.  
  **Alternatives considered:** Store feature id for every insert for audit replay.  
  **Why chosen:** Duplicate prevention is satisfied by `(station, minute)` under the agreed domain assumptions; omitting the column keeps Railway storage and ingest paths minimal until evidence says otherwise.

- **Decision:** **Include `ingested_at`** despite lean schema.  
  **Alternatives considered:** Only `observed_at` (sensor time).  
  **Why chosen:** Small column, high signal for “is the pipeline stale?” without retaining full raw JSON.

- **Decision:** **Defer `msc_id`** on station rows.  
  **Alternatives considered:** Store MSC now for symmetry with GeoMet catalog URLs.  
  **Why chosen:** Realtime joins and UI are **WMO-driven**; MSC is catalog convenience. Add when a feature needs it.
