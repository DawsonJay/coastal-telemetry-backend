## Technical Decisions and Trade-offs

- **Decision:** Default map uses most recent station reading (not synchronized timestamps).  
  **Alternatives considered:** Forcing all stations to same timestamp bucket (e.g., hourly sync).  
  **Why chosen:** Field teams need timely awareness; out-of-sync data with visible freshness is more useful than uniformly older data.

- **Decision:** Skip extra time-window/cadence filters in MVP map UI.  
  **Alternatives considered:** Multiple filter modes (minute/hourly presets) on day one.  
  **Why chosen:** Opacity-based freshness gives the key signal without additional controls, reducing cognitive load and implementation complexity.

- **Decision:** Keep long-range historical data retrieval on station detail page with chunked calls.  
  **Alternatives considered:** Attempting broad single-call history retrieval.  
  **Why chosen:** API pagination limits and payload size make chunking more reliable for one-year-scale exploration.
