## Technical Decisions and Trade-offs

- **Decision:** Reuse 15-minute logic from the range script by invoking the 15-minute script for each chunk.  
  **Alternatives considered:** Duplicating API/window logic directly inside range script.  
  **Why chosen:** Keeps behavior consistent between scripts and reduces drift risk in future changes.

- **Decision:** Anchor all chunk computations to one shared `now` point for range pulls.  
  **Alternatives considered:** Recomputing `now` for each iteration independently.  
  **Why chosen:** Prevents boundary skew during multi-call execution and makes exact-range trimming deterministic.

- **Decision:** Exclude rows lacking parseable timestamp fields from final trimmed range output.  
  **Alternatives considered:** Including untimestamped rows by default.  
  **Why chosen:** Exact-window contract depends on reliable timestamp comparison; unknown-time rows cannot be safely bounded.

## Problems Encountered

- **Problem:** Range chunk windows can drift if each chunk uses a slightly different "now" reference time.  
  **Diagnosis:** Multi-call workflows naturally introduce small execution delays between calls.  
  **Resolution:** Added a single anchor timestamp propagated through chunk calls so all window boundaries derive from one clock point.

- **Problem:** Exact-range trimming cannot be trusted for rows without parseable observation timestamps.  
  **Diagnosis:** Some rows may miss or contain invalid values in `obs_date_tm`/`date_tm-value`.  
  **Resolution:** Range output includes only rows with parseable timestamps that can be confidently verified against bounds.
